### 릴리안 웽의 "하네스 엔지니어링" — 모델 변경 없이 SWE-bench 20%→50%

- 오픈AI 전 안전 연구 부사장 릴리안 웽의 블로그 글 소개. 베이스 모델은 그대로 두고 모델을 둘러싼 실행 시스템("하네스": 계획·도구 활용·메모리·결과 평가 담당)만 개선해 SWE-bench 점수가 20%에서 50%로 상승
- 3가지 설계 패턴 제시: ① 워크플로 자동화(계획-실행-관측-수정 루프) ② 파일 시스템 기반 영속 메모리 ③ 서브에이전트를 통한 병렬 탐색
- 💡 **주목 포인트**: 같은 날 GeekNews 최상위 글("루프 엔지니어링의 미학", 43P)과 정확히 같은 주제 — "모델보다 하네스/루프 설계가 성능을 좌우한다"는 논지가 서로 다른 두 출처에서 동시에 확인됨.

### Claude Code 작성 코드 비중 급증 — 2월 10% → 5월 30~40%

- Claude Code 소속 Boris Cherny의 발언 인용: Claude Code가 실제로 작성하는 코드의 비중이 2월 10%에서 5월 30~40%로 급증. 2025년 겨울에는 100% 자동 생성 사례도 언급
- 💡 **주목 포인트**: 07-06 리포트의 "AI 코딩 시대 개발자 역할 변화" 논의를 뒷받침하는 실측 수치 — AI가 코드 작성의 절대량을 빠르게 흡수하고 있음을 보여줌.

### 안드레 카파시의 AutoResearch — 코드 경쟁력보다 "실험 반복 속도 설계" 능력

- 단일 GPU에서 5분 단위 학습을 반복하며 에이전트가 코드를 자동 수정하는 프로젝트 소개
- "앞으로 경쟁력은 코드를 얼마나 잘 짜느냐보다, AI가 실험을 더 빠르게 반복하도록 설계하는 능력으로 이동할 수 있다"는 전망 제시

## 2. HuggingFace 신규 VLM 업데이트 (경량 모델 — RTX 3070/4070 대응)

> 용도: 이미지를 보고 텍스트 질의에 응답하는 산업 안전 체크 (화재/연기 감지, 안전장비 착용 여부 등)

### 확인 결과 — 이번 조회에서도 프론티어급 신규 VLM 공식 릴리스 없음

- Hugging Face `image-text-to-text` 카테고리를 생성일 기준 최신순으로 확인. 상위 노출 항목(Qwen3.5-40B 계열 파인튜닝, GRM-2.6-Plus, gemma_4_e4b_bnb, gemma-3-4b, Qwen3.5/3.6 mlx 변환본 등)은 모두 개인/커뮤니티의 파인튜닝·양자화 버전이며, 공식 신규 베이스 모델 릴리스는 확인되지 않음
- YOLO, Ministral, SmolVLM 계열은 이번 조회 페이지에서 노출되지 않음 (기존 라인업 대비 변화 없는 것으로 추정)

### 선택 가이드 (누적, 변경 없음)

> - 초경량 단순 이진 질의 → **Qwen3.5-0.8B** 또는 **SmolVLM-256M/500M**
> - 화재/연기 위치 탐지 → **YOLO26** (실시간, 경량)
> - 복합 자연어 체크 → **Qwen3-VL 2B~4B** (양자화 필수), **Ministral-3-3B** 또는 **Gemma 4 E2B/E4B** (RTX 4070 12GB 대응)
> - 위치 기반 탐지 + 자연어 → **VLX-Seek**
>
> 지난 리포트들과 동일하게 이번 주에도 위 조합이 최선의 선택지로 유효함. 신규 프론티어 VLM 릴리스가 나오면 즉시 갱신 예정.
